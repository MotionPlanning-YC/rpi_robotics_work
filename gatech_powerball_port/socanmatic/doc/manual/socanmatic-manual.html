2. CANopen

CANopen is a communications protocol for embedded automation systems. In the OSI network model, CANopen covers the network, transport, session, and presentation layers (3-6).

2.1. Object Dictionary

Each CANopen node has an Object Dictionary, used by the master to read and write values such a status codes, physical parameters, and commanded actions.

Dictionary entries are addressed with a 16-bit index and 8-bit subindex. The subindex is used for compound object, e.g., records and arrays, in order to access the individual elements of that object.

Table 1. Dictionary Entry Ranges

Index RangeDescription
0x0000Unused
0x0001-0x001FSimple Data Types
0x0020-0x003FComplex Data Types
0x0040-0x005FManufacturer Specific Complex Data Types
0x0060-0x007FProfile-Specific Simple Data Types
0x0080-0x009FProfile-Specific Comple Data Types
0x00A0-0x0FFReserved
0x1000-0x1FFFComunications Profile Objects
0x2000-0x5FFFManufacturer Specific Objects
0x6000-0x9FFFStandardised Device Profile Objects
0xA000-0xAFFFStandardised Network Objects
0xB000-0xBFFFStandardised System Objects
0xC000-0xFFFFReserved
0x0000Unused

Entries in the dictionary are typed. Some common types include:

  • Boolean

  • INTEGER8

  • INTEGER16

  • INTEGER8

  • UNSIGNED16

  • UNSIGNED32

  • UNSIGNED32

  • REAL32

  • VISIBLE_STRING

  • OCTET_STRING

  • UNICODE_STRING

  • TIME_OF_DAY

  • TIME_DIFFERENCE

Data is encoded in little-endian format (least-significant byte first).

Glossary

CAN Object Identifier
(COB-ID)

11-bit identifier for a CAN frame.

Node

A (typically) embedded device performing some automation control, containing an Object Dictionary and communicating with the CANopen master.

Process Data Object
(PDO)

Message used to transmit real-time process data.

Service Data Object
(SDO)
